# Chapter 6: Picture and Model

## Core Idea

A good document pictures its subject by preserving relevant structure. Models, examples, diagrams, and explanations work only when their form fits the state of affairs they represent.

## Frameworks Introduced

- **Logical picture**
  - When to use: when building examples, diagrams, conceptual models, method descriptions, or research narratives.
  - How: decide which relations in the target must be preserved; remove features that distract from those relations.
- **Fit test for examples**
  - When to use: before using a toy example, analogy, or case study.
  - How: map the example's objects, relations, and constraints to the real case; mark any mismatches.
- **Model boundary note**
  - When to use: when an analogy or model may overextend.
  - How: state what the model captures and what it does not.

## Key Concepts

- **Picture**: a representation that shares relevant form with what it represents.
- **Fact / state of affairs**: what is the case; the configuration a document tries to represent.
- **Logical form**: the structural fit that lets a picture represent a possible situation.
- **Model**: a simplified picture used for reasoning or communication.
- **Misleading analogy**: a picture that preserves surface resemblance but breaks the important relation.

## Mental Models

- An example is not evidence unless it pictures the relevant structure.
- A diagram is useful only when its spatial relations correspond to conceptual relations.
- A method section should picture the actual procedure, not the idealized story of the procedure.
- A README should picture user action: install, configure, run, fail, recover.

## Anti-patterns

- **Surface analogy**: choosing an example because it feels similar, not because it shares the right structure.
- **Overfit model**: including details that do not matter to the target relation.
- **Unbounded metaphor**: allowing the model to imply more than it can support.
- **Decorative diagram**: visual structure that does not encode conceptual structure.
- **Case mismatch**: using an example whose constraints differ from the real use case.

## Key Takeaways

1. A representation must preserve the relation that matters.
2. Examples should be tested for structural fit.
3. Mark the boundary of models and analogies.
4. In scientific writing, distinguish evidence from illustrative picture.
5. In documentation, examples must match actual user paths.
6. In agent prompts, examples should picture desired behavior, not merely describe tone.
